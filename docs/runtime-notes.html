Observed model/backend/dtype results from phone testing. Use this as the source of truth before changing model runtime defaults.

Notes: current

Summary

Practical rule

On the tested Android phone/browser setup, SmolLM2 135M works on WebGPU/q4. SmolLM2 360M produces corrupt output on WebGPU q4/q4f16, but works on WASM/q8.

135M recommended: WebGPU + q4
360M recommended: WASM + q8
Avoid: 360M + WebGPU + q4/q4f16 on this device

Interpretation

What this means

The 360M model itself is not proven bad. The failure is specific to the tested browser runtime path. WebGPU/q4 and WebGPU/q4f16 produced corrupt generation for 360M, while WASM/q8 produced valid Python.

Do not treat “WebGPU available” as enough information. The tester needs model-aware defaults.

Patch guidance

Main tester default rules

If model is SmolLM2 135M:
  prefer WebGPU when available
  use q4

If model is SmolLM2 360M:
  prefer WASM
  use q8
  show warning if WebGPU is selected

For coding tests:
  use chat messages
  prefer temperature 0
  use 64 to 128 max new tokens
